Phase 5 walkthrough — Run loop & drivers

pyctrl Python front-end migration · the run-loop host, first hardware (core complete + live-verified 2026-06-04; AWG ports hardware-pending)

Goal of Phase 5: drive one real experiment end to end from Python and match MATLAB. Phases 0–4 made the front edge — value math, the seq tree, config/globals, and the ScanGroup parameter sweep — produce byte-identical blobs offline, with the engine never loaded. Phase 5 is the first phase that runs those blobs: it hosts the ZMQ job hub in the pyctrl process, consumes scan descriptors, compiles each point through the libnacs engine, and fires the FPGA / NI DAQ / camera on real hardware — then publishes frames to the same monitor and dashboard the MATLAB stack uses.

This doc walks the run-loop architecture (the three mutually-exclusive backend scenarios, the one ZMQ contract, the one descriptor currency), the entry point and the consume loop, the device-ownership re-scoping that collapsed "four drivers" down to essentially one real Python driver, the abort/pause/start control path (now memmap-free), the executable config, the scan→dashboard plumbing, code-snapshot replay, and the 2026-06-04 first-light hardware run. It closes with what is deliberately left for the hardware window and Phase 6.

TL;DR

  • The run loop runs. python -m launcher.run_loop.runner tcp://… hosts ExptServer.py in the pyctrl process and consumes scan descriptors over ZMQ. On 2026-06-04 it fired FPGA1, clocked out the NI DAQ, captured an Orca frame, and the dashboard showed the loading image with atoms loaded — the whole chain pyctrl→FPGA→Orca→relay→monitor→dashboard, end to end.
  • One contract, one currency. The monitor is a backend-agnostic ZMQ client; the hub (ExptServer.py) is the same code regardless of backend; the scan descriptor JSON is the only submission format pyctrl consumes. Because pyctrl is byte-identical (THE ONE RULE), a descriptor yields the same bytes whoever builds it.
  • "Four drivers" is really one. The FPGA (zynq) and the Siglent sequence output are engine-owned (just init_run→start→wait→post_run); the FPGA bytecode even generates the NI clock + start trigger. The only real Python driver in the run loop is NI DAQ clock-out (nidaqmx, listen-only). The Siglent/QICK recall paths + camera capture are sequence callbacks, armed out-of-band.
  • No memmap on the pyctrl side. Scenario 3 routes abort/pause/start through the ZMQ check_request gate directly — the file-based MemoryMap is a MATLAB-only artifact. Verified with a two-process integration test + a request-vs-reached paused ack.
  • Config is now code. pyctrl/expConfig.py is the live source, hot-reloaded per job; the frozen config_reference.json is demoted to a drift oracle. This resolved the config-snapshot staleness bug.
  • Status: the run-loop host is complete and live-verified. Remaining: Siglent + QICK AWG device ports are software-complete but hardware-unverified (need a downtime window + backend restart), and one open rearrange-scan SLM-lock wedge. See "Status & what's left."

The mental model — three scenarios, pyctrl is #3

The architecture rests on a user-ratified constraint: the two run loops (MATLAB and pyctrl) never run at the same time. That single fact deletes every hard problem a dual front-end would otherwise have — no hardware lease, no runtime arbitration, no two-writer precedence race, no concurrent-submission reconciliation. There are exactly three mutually-exclusive configurations:

ScenarioHub hostConsumerDescriptor→bytesControl path
1 · old monitor + MATLAB (production today)MATLAB procSequenceRunner.mdispatch_descriptor.m / ybStartScanmemmap (unchanged)
2 · new monitor + MATLABMATLAB procSequenceRunner.mdispatch_descriptor.mZMQ → server-side memmap bridge
3 · new monitor + pyctrlpyctrl procYbExptCtrl/runner.pydispatch_descriptor.pyZMQ check_request (no memmap)

Phase 5 builds scenario 3. The cost to MATLAB is ≈ zero: #1 is untouched, #2's descriptor path already existed, and in #3 MATLAB is simply off. A startup mutual-exclusion guard (assert_single_backend — port bind on the backend URL + a single-backend ping check) makes concurrency structurally impossible.

Two submission paths — and why pyctrl consumes only one

A scan can start two ways in the MATLAB world: a JSON descriptor (submit_scan_descriptor, the new monitor) or the ".m run button" (ybStartScan→submit_job, a MATLAB-proprietary byte stream needing a live MATLAB). Scenario 3 has no MATLAB, and the run-button payload is MATLAB-only — so pyctrl consumes the descriptor path only. handle_descriptor_pop dispatches a popped descriptor into a JSON job payload pyctrl emits and consumes (submit_job + link_descriptor_to_job), reusing the queue-UI linkage verbatim. A pyctrl-side ybStartScan (YbExptCtrl/yb_start_scan.py) gives MATLAB ergonomics by building a ScanGroup imperatively and exporting it to a descriptor — one payload format, no second intra-backend currency.

The run loop — entry point to inner loop

The launch path is deliberately layered so the host stays flat-importable and NO-HARDWARE-testable like the rest of the codebase:

python -m launcher.run_loop.runner tcp://127.0.0.1:1408
   │
   ├─ launcher/run_loop/runner.py   THIN shim: -m gives only the package root on sys.path,
   │                                so it prepends the flat pyproject pythonpath dirs
   │                                (lib, YbSteps, YbSeqs, YbExptCtrl, tools, + root for expConfig.py)
   │                                then delegates to:
   └─ YbExptCtrl/runner.py::main → serve(url)        the real host (port of SequenceRunner.m)

serve(url) wires the pieces and runs the loop:

serve(url):
    assert_single_backend(url)              # refuse to start if another backend is alive
    server  = ExptServer(url)               # host the ZMQ job hub IN this process
    seqcfg  = load_configs()                # SeqConfig.load_real() from expConfig.py + config.yml
    camera  = open_camera(seqcfg)           # pylablib OrcaCamera (NumImages>0 paths)
    run     = make_engine_run(server, camera, seqcfg)   # the per-job engine closure
    install SIGTERM/SIGINT handlers         # → clean DCAM release in finally
    consume_loop(server, should_stop, run_job_fn=…, dispatch_pop=…)

Inside consume_loop each iteration does three things, in order: pump camera commands (cooling/ROI/exposure/start-stop from the monitor), dispatch descriptors (handle_descriptor_pop turns any queued descriptors into jobs), and run the next job (run_job). When the queue is empty it falls to the idle scheduler (dummy/last-frame replay). The per-job flow:

handle_descriptor_pop(server)      # descriptor → submit_job(payload, summary) + link
run_job(server, payload)           # YbExptCtrl/sequence_runner.py
    build run-order + Scan.Params  # prep layer (ports ybBuildScanJob, NOT runSeq2 randperm)
    dispatch_descriptor(payload)   # → (seq_fn, ScanGroup)  inverse of the .m builder
    make_engine_run.run(seq, sg):
        for n in run-order:                       # the inner loop = ExpSeq(getseq(n)) → bytes
            seq = ExpSeq(sg.getseq(n))            #   (Phase 0–4 front edge, byte-identical)
            SequenceFn(seq); serialize(seq)
            init_run → pre_run → start → wait → post_run     # ENGINE drives the hardware
            (callbacks fire: AWG recall, camera capture)
            check_request()                       # abort / pause / start gate, between shots

The inner loop is the same shape as the MATLAB runner, and the front-edge (everything up to serialize) is exactly the Phase 0–4 code already pinned byte-for-byte. Phase 5 adds only what surrounds it: the hub, the dispatch, the engine calls, the device touches, and the control gate.

Device ownership — why "four drivers" is really one

The headline scope ("FPGA + NI + Siglent + QICK + camera") reads like five drivers. Verified against lib/ExpSeq.m, config.yml, and the libnacs C++ zynq/nidaq backends, the run loop touches hardware in exactly one place (run_bseq), and the touch is of three kinds:

KindDevicesWhat pyctrl does
Engine-owned
(zero front-end driver)		 FPGA1 (zynq) · Siglent AWG1 sequence output Nothing device-specific — they ride out on init_run → pre_run → start → wait → post_run. The FPGA bytecode also generates the NI DAQ sample clock (400 kHz on a dedicated CLOCK line) and the start trigger (FPGA TTL0, reserved). Wiring: FPGA clock → NI PFI0; FPGA TTL0 → NI PFI1.
Front-end-owned
(the one real driver)		 NI DAQ clock-out (PCIe-6738) devices/nidaq/nidaq_runner.py — a fresh port onto the nidaqmx package. It does not generate the clock; it configures the card to listen (cfg_samp_clk_timing(source="/Dev1/PFI0", FINITE) + cfg_dig_edge_start_trig("/Dev1/PFI1", RISING)), transposes, writes, starts, waits.
Callback-invoked
(armed out-of-band)		 Siglent recall · QICK FPGA_AWG · camera capture Sequence callbacks (regBeforeStart / regAfterEnd), TTL-triggered. The run loop only needs to invoke the registered callbacks; the callback-registration infra was already ported (exp_seq.py reg_before_start, root_seq.py before_bseq_cbs).

The #1 silent-bug risk: MATLAB queues sample-major [nsamps, nchns]; nidaqmx writes channel-major [nchns, nsamps] — a transpose is required and there is no error if you get it wrong (the card just clocks out garbage). Verified non-degenerate both compile-only and live (14 ch × 1048 samps, every column). The clock rate is deliberately set to 500e3 as an over-estimate of the real 400 kHz — an under-estimate makes the card stop early.

Control — abort / pause / start without memmap

In scenarios 1–2 the control flags live in a file-based MemoryMap that MATLAB polls. Scenario 3 has no MATLAB and no memmap: the monitor sends ZMQ verbs (pause_seq / abort_seq / start_seq), the hub records them, and the inner loop reads them through the check_request gate between shots. The monitor side is backend-aware: memmap_signal.mmap_open(backend) returns None for non-MATLAB backends, so the Tkinter panel falls through to ZMQ and the dashboard branches on YB_BACKEND.

  • Abort stops the run-order loop and finishes the job; a stale-abort wedge (a leftover Abort flag silently making the next scan run 0 iterations) is fixed by clearing all control flags at job start.
  • Pause is request-vs-reached: the loop acks when it has actually reached the paused state, not merely received the request, so the monitor can distinguish "Pausing…" from "Paused."
  • An abort-while-idle gate keeps a pause/abort sent to an empty queue from poisoning the next job.

Verified with a two-process integration test (one process runs the loop, another sends ZMQ control) and unit tests for the reached-paused ack and the clear-at-job-start fix.

Known open item: get_status coarseness — pyctrl reports a single ZMQ status string; the rich Pausing…/idle-dummy sub-states were memmap-only and remain an optional future get_status_rich verb. (Separately, a now-resolved bug had get_status stuck on "running" after a finite scan — fixed via mark_idle_if_queue_empty, live-verified 2026-06-05.)

Config — executable expConfig.py, hot-reloaded

Phases 0–4 fed off a frozen JSON config snapshot. Phase 5 made the config code: pyctrl/expConfig.py is the twin of matlab_new/expConfig.m and the live source. SeqConfig.load_real() builds from it, and the runner hot-reloads it per job (load_real(reload=True), preserving SeqConfig.G) so a recalibration edit goes live without a backend restart. The engine config.yml was copied into the submodule too, so the front-end is self-contained.

The committed config_reference.json is demoted to a drift oracle: tests/test_exp_config.py asserts expConfig.py ⇔ snapshot and fails loudly on drift, and the per-point byte oracles are a second net. This resolved the config-snapshot staleness bug. Editing policy (2026-06-05): expConfig.py is the source of truth — recalibrate it, regenerate config_reference.json via tools/capture_config_reference.py (no MATLAB), then sync expConfig.m and re-capture the MATLAB byte oracles to keep byte-equality (THE ONE RULE).

Scan → dashboard plumbing

So the new monitor shows the same scan metadata the MATLAB stack does, the runner emits three things, all from lib/scan_summary.py + lib/scan_prep.py:

Beyond MATLAB, the baseline SeqConfig.get().consts is embedded as config['expConfig'] so a scan is fully reproducible (effective = expConfig ⊕ ScanGroup overrides ⊕ sweep point) — MATLAB saved only the user deltas. Production run-order — scramble-within-pass + stack

pyctrl randomizes like MATLAB production (ybBuildScanJob), NOT runSeq2's global randperm. The run-order construction lives in the prep layer (lib/scan_prep.py): stack(), scramble_groups() (shuffle within each consecutive block, boundaries intact), and build_scan_order(nseqs, *, stack_num, scramble, rng) → StackNum back-to-back passes over 1..nseqs, each pass independently shuffled when scrambling. The scan loop (run_seq.py) is unchanged — it just runs the order it is handed.

KnobValue
passes (StackNum)explicit rep opt (≥1, bypasses the formula) else max(ceil(NumPerGroup/nseqs), 2) (NumPerGroup default 200)
scrambleDEFAULT OFF, runp-only: bool(runp.Scramble, default 0) — on only when the scan sets g.runp().Scramble = 1. The random opt does NOT control per-pass scramble (it only selects runSeq2's global shuffle on the rep=0 forever path).

Note: not byte-reproducible vs MATLAB's randperm — it is behavior-faithful (the per-block invariant matches; the realized shuffle does not).

Per-run code snapshot + replay

Every scan records what code ran (mirroring the SLM server's code_snapshot.py). scan_prep.write_scan_config calls snapshot_code() (best-effort; kill-switch YB_CODE_SNAPSHOT=0): content-addressed blobs at <DATA>/_code_snapshots/<sha>.py (write-once dedup) + a readable per-run tree at _runs/<scan_id>/ (hardlinks at original rel paths) + manifest.json (hashes, roles, git state). Roles are the seq_reload boundary: YbSeqs/YbSteps/YbScans/YbRearrangement = experiment (replayable); lib/expConfig/config.yml/YbExptCtrl = record-only (swapping framework classes breaks isinstance/singletons — only lib_mismatch()-detected, never replayed).

Replay is opt-in: a descriptor may carry code_snapshot: {scan_id: N} (the dashboard's "re-queue + code"). run_job wraps its dispatch + run in snapshot_syspath(...) — prepends that run's snapshot experiment dirs to sys.path, evicts cached experiment modules, runs, restores. Absent field → nullcontext → byte-identical to before, so every normal run is untouched.

Camera capture — pylablib + the wire format

Production capture is MATLAB IMAQ (videoinput('hamamatsu') in OrcaImag.m); ExptServer.py is only the image relay (a mailbox). pyctrl therefore captures via pylablib (devices/orca/orca_camera.py) and feeds the hosted relay (lib/frame_capture.py store_imgs). Validated on the real C15550-20UP: internal snap, external TTL-triggered capture (after fixing a rising-edge polarity gotcha), MATLAB-IMAQ-parity start_video/stop_video/frames_available/ flush/is_running + cooling/ROI/exposure. Trigger confirmed FPGA1/TTL54 (not TTL14 = QICK).

store_imgs's wire format is a column-major flatten + 3-element shape prefix packed MATLAB-side — the pyctrl capturer must reproduce it or yb_analysis mis-reads the frames. ⚠ get_num_imgs reads 0 because get_imgs resets nseq_imgs; confirm capture out-of-band via get_seq_num deltas instead.

The pre-flight harness (tools/)

To de-risk before any start(), the harness runs compile-only codegen checks first:

  • preflight_hwtest1.py — FPGA byte/codegen, all hardware-free.
  • preflight_ni.py — NI transpose + dump-enabled clock read-back.
  • preflight_camera.py — camera path.
  • submit_hwtest1.py / submit_ni.py / submit_camera.py — dry / --fire descriptor submitters.
  • query_backend.py — status/queue probe.
