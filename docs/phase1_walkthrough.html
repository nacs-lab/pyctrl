1. What Phase 1 builds

Phase 0 proved we can read the byte blob MATLAB hands the engine. Phase 1 builds the half that produces it: the value algebra (SeqVal) and the node table + serializer (SeqContext). When you write an expression like g(0) * 1000 in Python, these two classes turn it into the exact same numbered node graph and bytes the MATLAB stack emits.

Python expression  # e.g. g0 * m1, or round(g0 * 1000)
        │  SeqVal operator overloading        # lib/seq_val.py
        ▼
build-time expression tree (head + args)
        │  SeqContext.get_val_id / ensure_serialize   # lib/seq_context.py
        ▼
numbered node table  ──►  node_serialized() bytes  ── must equal MATLAB ──┐
        ▲                                                                   │
        └────────────── compare_bytes.decode() (Phase 0 reader) ───────────┘
THE ONE RULE, restated for Phase 1 For any value graph pyctrl builds, the serialized node / data / global tables must be byte-identical to MATLAB's. A SeqVal is a pure build-time AST — it is never evaluated in Python; the libnacs engine evaluates it after deserialization. So Phase 1 is entirely NO-HARDWARE: it is byte math, checked against MATLAB output, safe to run during a live experiment.

2. SeqVal — the value AST

A SeqVal is one node of an expression tree: a head (an opcode, or a negative tag for a leaf), a list of args, the owning ctx, and a node_id (0 until serialized). Arithmetic is built by overloading Python's operators, so ordinary syntax grows the tree:

e2 = g0 * m1          # SeqVal(OP_MUL, [g0, m1], ctx)
e5 = interpolate(arg0, e2, e3, [1,2,7,3,4,5])   # SeqVal(OP_INTERP, [arg0, e2, e3-e2, data], ctx)
v  = round(g0 * 1000) # SeqVal(OP_RINT, [SeqVal(OP_MUL, [g0, 1000.0])])

Leaves vs operations

Three leaf kinds carry a negative head so they can never collide with an opcode: H_ARG = -3 (arg(i), a sub-sequence parameter), H_MEASURE = -1 (m(id), a value measured mid-sequence), H_GLOBAL = -2 (g(i), a runtime-settable slot). Everything else is an operation whose head is an opcode 1–50. Constants are not SeqVals at all — a bare Python float/int/bool rides directly as an arg.

Construction-time constant folding

MATLAB folds trivial expressions as it builds them, and the byte output depends on it, so the port reproduces it exactly:

g0 + 0   → g0          0 * g1  → False      g0 - g0 → 0
g0 ** 2  → g0 * g0     g0 ** 0 → int32(1)  1 ** g1 → 1
xor(g0, 1) → ~g0       ifelse(True, g0, g1) → g0

These short-circuits are why the equality check used during folding, seqval_isequal(a, b), is a structural recursion (head + args, type-insensitive for numerics) — it must mirror MATLAB's isequal, not Python identity. (Note g0 - g0 folds to the double 0, but g0 * 0 folds to the logical False — the same distinction MATLAB makes, and it matters for which constant tag is serialized.)

No local evaluator (by design) pyctrl never computes the numeric value of a SeqVal — the engine does, every shot, because globals and measures are unknown at build time. So seq_val.py is a pure build-time AST. (This trivially satisfies the build/eval split the plan borrows from brassboard-seq's rtval: there is simply no evaluator to split off.)

3. The type mapping

The serialized constant tag (bool / int32 / float64) is chosen by the Python type of the value, mapped to mirror MATLAB's isfloat / islogical / isinteger dispatch. The one surprise: a bare Python int is a float64, because a bare MATLAB literal (e.g. 3) is a double. To get an int32 constant you use a numpy integer, exactly as MATLAB needs int32(3).

Python valueSerialized asMATLAB equivalent
float / numpy.floatingfloat64 (ArgConstFloat64)1.3, double(x)
int (bare Python int)float643 (a bare literal — a double)
bool / numpy.bool_bool (ArgConstBool)true / false
numpy.integer (int8/16/32/64, uint*)int32 (ArgConstInt32)int32(3), int8(23)

All numpy integer widths intern as the same int32 constant (int8(23) and int16(23) share one node), matching MATLAB's int32(val) in getValID. The single helper _arg_kind() performs this classification and is used by both serialize_arg and get_val_id so they can never disagree.

4. SeqContext — interning & the serializer

SeqContext is the per-sequence bookkeeping table. It mints the leaf factories (get_arg, new_measure, new_global), interns constants, and — the heart of Phase 1 — serializes nodes into the byte layout. The byte format is the value layer of the Phase-0 format: [nnodes:4B][ node record × nnodes ], each record [opcode:1B][ arg × arity ] with self-describing args [argtype:1B][payload].

The two entry points

  • get_val_id(val) — intern a value as a top-level node and return its 1-based id. Constants get an OP_IDENTITY wrapper node (and are de-duplicated); a SeqVal is serialized if it has no id yet.
  • serialize_arg(res, arg) — append one argument to a node being built. Leaves (arg/measure/global) and constants are written inline; a composite SeqVal is written as ArgNode + the id of its node (serializing it first if needed). This inline-vs-reference split is what makes the table a shared DAG.

One naming collision worth knowing

MATLAB has both a private property node_serialized (the storage cell array) and a method nodeSerialized(). They snake_case to the same name, so the port stores the list as self._node_serialized (it is Access=private in MATLAB anyway) and keeps the method as node_serialized().

Little-endian everywhere Every multibyte field is packed with struct '<' to match MATLAB typecast(...,'int8') on x86. Wrong byte order is the single most common serialization bug, so test_serialize.py pins it directly (e.g. 1.0 → 00 00 00 00 00 00 f0 3f).

5. Worked example: build → bytes

This is the real node table from the TestSeqContext port — the same construction MATLAB's dotest builds. The Python below produces the byte table on the right (shown verified against MATLAB in §7).

count (uint32) opcode arg type payload 
ctx = SeqContext()
arg0 = ctx.get_arg(0);  m1, _ = ctx.new_measure()
g0, _ = ctx.new_global(SeqVal.TYPE_BOOL)
e2 = g0 * m1                      # OP_MUL(global 0, measure 1)
ctx.get_val_id(1.3)               # node 1: const float64
ctx.get_val_id(True)              # node 2: const bool
ctx.get_val_id(np.int8(23))       # node 3: const int32
ctx.get_val_id(e2)                # node 4: g0 * m1

The first four nodes, byte by byte

0f 00 00 00 nnodes = 15 (the full dotest graph)
32 02 cd cc cc cc cc cc f4 3f node 1 = identity( float64 1.3 ) — 0x3ff4cccccccccccd, little-endian
32 00 01 node 2 = identity( bool true )
32 01 17 00 00 00 node 3 = identity( int32 23 ) — 0x17
03 05 00 00 00 00 04 01 00 00 00 node 4 = mul( global #0 , measure #1 ) = g(0) * m(1)

Node 4 shows the inline-leaf rule: g0 and m1 are leaves, so they are written straight into the OP_MUL record as ArgGlobal/ArgMeasure — no separate rows. A composite operand (say a nested e4 = e3 - e2) would instead appear as ArgNode + its id, which is how (g(0)*2)*1000-style trees become a shared graph. OP_IDENTITY (50) is the uniform wrapper for a bare constant that needs its own id.

Interning & de-duplication

Three separate caches make each distinct constant appear once: const_f64_ids (keyed by the 8 packed bytes, so it splits +0.0/-0.0 like MATLAB's hash), const_i32_ids, and const_b_ids. Interpolation tables are de-duplicated the same way in get_data_id — the dotest's two interp calls that share [1,2,7,3,4,5] reference one data entry, exactly as MATLAB packs them.

6. Comparison operators & canonicalization

One Python-specific wrinkle is worth its own section, because it shapes how comparisons are written and verified. Python has no __rlt__: a constant on the left of a comparison reflects to the opposite operator on the right operand.

3 < g     # Python evaluates int.__lt__(3, g) -> NotImplemented
          # then g.__gt__(3)  =>  GT{g, 3}

3 < g     # MATLAB: lt(3, g)  =>  LT{3, g}

So the front-end serializes GT{g,3} where MATLAB serializes LT{3,g}. These are the same comparison (3 < g ≡ g > 3, true even under IEEE-754 NaN, where both are false) — but the bytes differ: a different opcode and swapped args. Arithmetic does not have this problem; its reflected dunders (__radd__ …) preserve operand order, so 3 - g serializes identically to MATLAB.

The decision: keep operators ergonomic, make the comparator smart Rather than force porters to write lt(3, g) instead of 3 < g (tedious and un-Pythonic), the byte comparator canonicalizes swappable comparison nodes. So 3 < g is fine to write; verification treats the reflected form as equivalent.

compare_bytes.py grew canonical_node() / normalize(): orderings are expressed as LT/LE (flip GT→LT, GE→LE and swap args), and the commutative EQ/NE have their two args sorted. After that, the two forms are equal:

Python 3 < g0 → GT{g0, 3}:
06 05 00 00 00 00 02 … 08 40
MATLAB 3 < g0 → LT{3, g0}:
05 02 … 08 40 05 00 00 00 00

canonical_node(GT{g0,3}) flips to LT{3,g0} → the two match. test_compare_canonical.py proves this end to end for < > <= >=.

Narrow and safe — not a blanket relaxation Reflection only fires with a constant on the left, so at most one operand is a compound sub-node and the node-graph ids stay aligned — canonicalizing the comparison node alone is sufficient. A genuine opcode mistake (e.g. GT where LT was meant, same arg order) still diffs. compare_bytes.py … --strict requires literal byte-identity. THE ONE RULE is now "byte-identical up to comparison-operator orientation."

7. Proving byte-equality

Phase 1 correctness rests on three checks, strongest first:

  1. Ported reference bytes. test_seq_context.py reproduces TestSeqContext.m, asserting the exact node_serialized() / data_serialized() / global_serialized() bytes the MATLAB test pins.
  2. Live MATLAB still emits those bytes. Running the MATLAB suite on this machine — TestSeqContext 5/5, TestNumToStr 5/5, TestMath 4/4 — confirms the reference literals are what MATLAB produces today.
  3. Direct hex diff (transcription-free). The same dotest graph was built in live MATLAB and in Python and the raw tables compared — byte-identical:
# MATLAB  tohex(ctx.nodeSerialized())   vs   Python  ctx.node_serialized().hex()
NODES  : MATCH   (171 bytes)
DATA   : MATCH   (100 bytes)
GLOBAL : MATCH   (6 bytes)
RESULT : ALL THREE TABLES BYTE-IDENTICAL (MATLAB == Python)

The first byte of NODES is the 0f 00 00 00 (15 nodes) seen in §5; the DATA table is the two shared interpolation vectors; GLOBAL is 02 00 00 00 01 03 — two slots, types bool(1) and float64(3).

8. The SeqPlotter .seq addition

Alongside the serializer sits a second, output-only format: .seq, the flattened sequence that SeqPlotter reads to draw what each channel actually does over time. It is not serialize():

serialize() (§1–7).seq (this section)
Contentssymbolic node graph + tablesevaluated per-channel (time, value, pulse_id) points
Consumerthe libnacs engine (compiles it)SeqPlotter (a Dash/Plotly viewer)
Who computes itbuilt by SeqVal/SeqContextthe engine — get_nominal_output(pts_per_ramp)
Bound by THE ONE RULE?yes — fed to the engineno — a viewer artifact, never fed back

What pyctrl does (and does not) do

Crucially, pyctrl does not re-evaluate the sequence in Python. The engine already produces the points; lib/dump_output.py only orchestrates the engine call and packs the result, delegating the actual byte layout to tools/compare_seq_bytes.py so reader and writer share one source of truth:

def dump_output(engine_seq, pts_per_ramp=100, seq_name="", params=None, inverse_chn_map=None):
    res = engine_seq.get_nominal_output(pts_per_ramp)   # engine evaluates
    channels = channels_from_nominal_output(res, inverse_chn_map)
    return pack([build_seq_struct(seq_name, 1, channels, params)])   # compare_seq_bytes.encode

dump_output() handles a single basic sequence and is compile/eval-only (safe on a compiled handle). dump_output_branches() walks init_run → pre_run → get_nominal_output → post_run for multi-branch sequences — init_run is NEEDS-HARDWARE, so it runs only with the real engine in a downtime window (the board-free dummy raises on it by design).

The .seq byte format

Verified by an exact round-trip of a real 132,855-byte MATLAB sample (tests/reference/seqplotter_sample_ryddet.seq). Per point: [time int64][value float64][pulse_id uint32] = 20 bytes, matching SeqPlotter's struct.unpack('<qdI', …).

[nseqs : uint32]
per seq:
  [seq_name\0]                 # "yyyymmdd_HHMMSS:name" — the only nondeterministic field
  [seq_idx : uint32]           # 1, then 2.. for branch targets
  [nchns : uint32]
  per channel:
    [chn_name\0]               # alias-decorated "alias1, alias2 (nominal)" (Phase 3 map)
    [npts : uint32][ time i64, value f64, pulse_id u32 × npts ]
  [has_params : uint8][ params\0 ]   # JSON; present iff has_params
[has_bt_info : uint8][ … ]     # optional debug backtrace; not needed for plotting

The highlight hook for scanned parameters

Each params leaf is {"value", "type", "config_value"?} — there is no native "this was scanned" flag. Because .seq is a viewer artifact (not bound by THE ONE RULE), pyctrl is free to enrich it: mark_scanned(params, scanned_paths, scan_dims) injects "scanned": true (and optionally "scan_dim") into the swept leaves, which the dashboard reads to highlight them. A strict-parity .seq simply omits the call.

Where it lands later The seam is ready now; the remaining wiring is mechanical and phase-gated: alias-decorated channel names and the params-JSON block need the channel map + convert_seqval_to_string port (Phase 3); the per-scan-point scanned marker needs ScanGroup.get_seq(n) (Phase 4); auto-save on each run + the dashboard view come with the run loop (Phase 5–6). See docs/seqplotter_integration.md.
