1. Overview & data flow

The whole point of Phase 0 is to nail down the one interface between the MATLAB front-end and the C++ engine: the byte array that ExpSeq.serialize() produces and Manager.create_sequence() consumes. If a Python program can read those bytes back into a faithful structure and re-emit them byte-for-byte, it understands the format completely.

MATLAB ExpSeq.serialize()  ──►  bytes  (.bin or seqN.json)
                                  │
                  compare_bytes.decode()      # the reader
                                  ▼
                  nested Python structure (dicts / lists)
                                  │
                  compare_bytes.encode()      # the exact inverse
                                  ▼
                                bytes        ── must equal the original ──┐
                                  ▲                                       │
                                  └───────────────────────────────────────┘
Why this is safe to run anytime The references already exist on disk (committed by the MATLAB test suite, captured engine-free). The reader is pure standard-library byte math in a separate Python process — it never imports libnacs and never opens a device. Zero MATLAB execution, zero hardware, zero interference with a running experiment.

2. The byte format

Two layouts, both documented in matlab_new/lib/SeqContext.m. All multi-byte values are little-endian (MATLAB typecast(...,'int8') on x86).

Top level

[version=0 : 1B]
[nnodes : 4B][ node record × nnodes ]              # the value / expression graph
[nchns : 4B][ NUL-terminated channel name × nchns ]
[ndefvals : 4B][ chnid:4B, type:1B, value:1–8B × ndefvals ]
[nslots : 4B][ type:1B × nslots ]                  # global variable slot types
[nnoramp : 4B][ chnid:4B × nnoramp ]               # usually 0
[nbasicseqs : 4B][ basic sequence × nbasicseqs ]
[ndatas : 4B][ ndouble:4B, data:8B×ndouble × ndatas ]   # interpolation tables
[nbackenddatas : 4B][ devname\0, size:4B, data:size × … ]  # "ZYNQZYNQ" device blocks

Each basic sequence

[ntimes : 4B][ sign:1B, id:4B, delta_node:4B, prev_id:4B × ntimes ]
[nendtimes : 4B][ time_id:4B × … ]
[ntimeorders : 4B][ sign:1B, id:4B, before_id:4B, after_id:4B × … ]
[noutputs : 4B][ id:4B, time_id:4B, len:4B, val:4B, cond:4B, chn:4B × … ]   # pulses
[nmeasures : 4B][ id:4B, time_id:4B, chn:4B × … ]
[nassigns : 4B][ assign_id:4B, global_id:4B, val:4B × … ]
[nbranches : 4B][ branch_id:4B, target_id:4B, cond:4B × … ][ default_target:4B ]

A node record

Uniformly [opcode:1B][ arg × arity(opcode) ], and each argument is self-describing: [argtype:1B][payload]. The "standalone" constant / global / measure / arg nodes are not special — they are simply OPIdentity (50) nodes with one argument. The only exception is OPInterp (48): three args plus a trailing data_id:4B.

Arg payload sizes: bool=1B, int32=4B, float64=8B; node / measure / global / arg are 4-byte uint32 ids. Full opcode and arg-type tables are in the appendix.

3. Worked example: decoding a whole sequence

The smallest reference, seq3_waitfor.json (174 bytes, 3 nodes). The MATLAB file even ships with // comments, so we can check the decoder against the original author's own annotations. Below: the raw bytes, color-coded, then the structure the reader produces.

version count (uint32) opcode arg type payload channel string table record

Header + the three value nodes

0 version
3 0 0 0 nnodes = 3
50 2 0 0 0 0 0 0 0 0 node[0] = identity(float64 0.0)
3 5 0 0 0 0 2 0 0 0 0 0 0 0 64 node[1] = mul( global #0 , float64 2.0 )
3 3 2 0 0 0 2 0 0 0 0 0 64 -113 64 node[2] = mul( node #2 , float64 1000.0 )

How the two non-trivial bytes decode

The float 1000.0: the 8 payload bytes are 00 00 00 00 00 40 8F 40 (note -113 is the signed-int8 spelling of 0x8F = 143). Read little-endian that is the IEEE-754 double 0x408F400000000000 = 1000.0. Likewise 2.0 is 0x4000000000000000 and 0.0 is eight zero bytes.

The channel name: later in the file the bytes 68 101 118 105 99 101 48 47 67 72 57 0 are ASCII D e v i c e 0 / C H 9 + NUL → "Device0/CH9".

Header + channels + globals (continued)

1 0 0 0 nchns = 1 68 101 118 105 99 101 48 47 67 72 57 0 "Device0/CH9"
0 0 0 0 ndefvals = 0
1 0 0 0 3 nslots = 1, type 3 (float64) — one global
0 0 0 0 nnoramp = 0
1 0 0 0 nbasicseqs = 1

...which the reader turns into this (abridged)

Reading the nodes back as an expression: node[1] = g(0) * 2 and node[2] = node[1] * 1000 = (g(0) * 2) * 1000 — the wait-for duration, recovered exactly from the bytes.

4. The value / expression graph

The nodes section is the most important idea in the format: a flat, numbered table that encodes a directed acyclic graph of expressions — essentially SSA form. Parameters in this system are not plain numbers; they are lazy expressions that may depend on runtime quantities. A pulse amplitude might be g(0) * 1000; a wait might be round(g(0) * 2 * 1000); a branch might fire on m(3) < 0. Each is a small tree, and all of a sequence's trees are flattened into one shared, numbered node table.

Every row is one of two things — a leaf (an input to the computation) or an operation (combining other nodes, its arguments).

The four kinds of leaf

LeafMeaning at runtimeIn the bytes
constant (bool/int32/float64)a literal valueArgConstFloat64 + 8B, etc.
global g(i)a runtime-settable variable (slot i); set_global(i,v) changes it between shotsArgGlobal + slot id; the slot's type is also listed in the top-level slots table
measure m(id)a value measured during the sequence (e.g. atom detection) that later expressions / branches can useArgMeasure + measure id
arg arg(i)a parameter passed into a sub-sequence — like a function argumentArgArg + index

Globals, measures, and args are the whole point of having a graph: they are unknown when the sequence is built, so the value must be carried symbolically and evaluated later.

How it is built (MATLAB side)

SeqVal overloads the arithmetic operators, so writing g + 2 or round(g*1000) constructs SeqVal(opcode, {args}) nodes — an expression tree grown straight from ordinary MATLAB syntax. SeqContext is the interning table that flattens those trees into the numbered, de-duplicated node list at serialize time.

The key encoding rule: inline leaves, reference sub-expressions

Not every sub-value gets its own row. From serializeArg:

  • Leaves are inlined directly into the parent operation's argument list.
  • Composite sub-expressions are referenced by id — emitted as ArgNode + the id of another node in the table. This is what creates the shared DAG.

Both appear in the verified seq3 graph for (g(0)*2)*1000. Compare node 2 (two inline leaves) with node 3 (a reference + an inline leaf):

node 2 = mul( g(0) , 2.0 ):
03 mul 05 00 00 00 00 arg = global, slot 0 (inline leaf) 02 00 00 00 00 00 00 00 40 arg = float64 2.0 (inline leaf)
node 3 = mul( node#2 , 1000.0 ):
03 mul 03 02 00 00 00 arg = NODE, id 2 → references node 2 02 00 00 00 00 00 40 8F 40 arg = float64 1000.0 (inline leaf)

The only structural difference is the first argument: an inline global leaf in node 2 versus an ArgNode → 2 reference in node 3. That reference is what links (g(0)*2) into (g(0)*2)*1000.

Why there are no special "leaf records" When a bare leaf needs its own id — because something references it by id (e.g. a pulse length that is just a constant, or the 0.0 time base in node 1) — it is stored as an OPIdentity (opcode 50) node wrapping the single leaf. So every row is uniformly [opcode][args]; a referenced constant is simply identity(0.0). getValID / ensureSerialize mint these ids and intern constants, so each distinct literal appears once — that interning is what gives the graph its DAG sharing.

The rest of the sequence points into the graph

Almost every other table refers to the value graph by 1-based node id (0 = none / absent):

TableField(s)Meaning
timesdelta_nodethe time offset is itself an expression
outputs (pulses)val, len, condamplitude/ramp, duration, gating condition (cond=0 = unconditional)
assignsvalnode id written into a global
branchescondnode id deciding control flow

The graph is the shared "value layer" that the timing and control layers compute against.

Why serialize a graph instead of numbers The engine JIT-compiles this DAG with LLVM. Because globals and measures are symbolic, the engine recomputes the graph every shot — pre_run re-evaluates it for the current global values, and a measurement taken mid-sequence flows into a later cond to drive a branch — without rebuilding or recompiling the sequence. The numbers frequently do not exist yet at build time, so they must be carried as expressions.

Worked example: seq1's expression

seq1's 15-node graph encodes the value TestExpSeq builds, whose human form is:

(4600 + ifelse(m(3) < 0, 3400, 0) + round(g(0) * 1000)) / 1000

The first nodes read straight out of the table:

node 1 : mul( global #0 , float64 1000.0 )   # g(0) * 1000
node 2 : rint( node #1 )                      # round(g(0) * 1000)
node 3 : cmpLT( ... )                          # m(3) < 0   (opcode 5)
  ...                                          # select/ifelse, adds, div → the full tree

A measurement (m(3)) and a global (g(0)) feed arithmetic and a comparison; the comparison feeds an ifelse (OPSelect); the results are summed and divided — exactly the tree the round, ifelse, and / 1000 in the MATLAB produced. Phase 1 ports this first: SeqVal + SeqContext must emit a byte-identical node table, checked with compare_bytes.decode against these references.

5. compare_bytes.py — the reader, part by part

The constant tables = the format spec

Three lookup tables, transcribed from matlab_new/lib/SeqVal.m, encode everything needed to walk the bytes:

  • _ARITY — opcode → number of arguments. Essential, because node records are variable length: to find where one node ends you must know its arg count.
  • _OP_INTERP = 48 — the lone special case (3 args + a 4-byte data_id).
  • _ARG_NAME / _ARG_CODE — arg-type tag ↔ name (decode & encode).

The cursor

A small class holding (bytes, position). Every read advances the cursor and bounds-checks; endianness lives in exactly one place.

class _Cur:
    def take(self, n):
        if self.i + n > len(self.d):
            raise ValueError("overrun at byte %d" % self.i)
        b = self.d[self.i:self.i+n]; self.i += n; return b
    def u32(self): return struct.unpack("<I", self.take(4))[0]   # little-endian
    def f64(self): return struct.unpack("<d", self.take(8))[0]
    def cstr(self): # read up to the NUL, then skip it ...

Decoding one node

This is the crux. Read the opcode, look up its arity, read that many self-describing args, and (only for OPInterp) one more 4-byte field:

def _decode_node(c):
    op = c.u8()
    node = {"op": op, "args": [_decode_arg(c) for _ in range(_ARITY[op])]}
    if op == _OP_INTERP:
        node["data_id"] = c.u32()
    return node

def _decode_arg(c):
    t = c.u8()
    if t == 2: return {"argtype":"float64", "val": c.f64()}   # 8B
    if t == 1: return {"argtype":"int32",   "val": c.i32()}   # 4B signed
    ...                                                  # node/measure/global/arg = u32

Walking the earlier example: byte 3 is OPMul, _ARITY[3]=2, so two args follow — a global and a float64. Byte 50 is OPIdentity, arity 1, one arg. No special cases.

Top-level decode & the strict consume check

decode() reads the sections in order and ends with:

if c.i != len(c.d):
    raise ValueError("trailing %d bytes after decode" % (len(c.d) - c.i))

This is what makes round-trip a strong test: if any section size is off by even one byte, decode either overruns or leaves trailing bytes — and the test fails loudly rather than silently mis-parsing. (One subtlety: the int8([0,0,0,0]) you see in ExpSeq.serialize is the noramp count of 0, not padding.)

load() — getting bytes in

def load(path):
    if path.endswith(".json"):
        text = re.sub(r"//[^\n]*", "", open(path).read())  # strip // comments
        text = re.sub(r",\s*]", "]", text)               # tolerate trailing comma
        return bytes((x+256)&0xFF if x<0 else x for x in json.loads(text))  # int8→ubyte
    return open(path, "rb").read()

The comment-stripping mirrors MATLAB's own read_json; the int8 → unsigned mapping is why -113 in the file becomes byte 143.

6. Pulses: from MATLAB code to bytes and back

The most satisfying check is to follow a real pulse round-trip. seq1 is built by this MATLAB (from TestExpSeq.m):

s.addStep(1).add('Device1/CH1', 4);                     # a value pulse
s.conditional(true).addStep(...).add('Device2/CH3', -1);
m1 = s.addMeasure('Device2/CH5');                       # a measurement
s.addStep(1.2).add('Device1/CH5', @(t) t - 2.3);        # a ramp

The reader extracts these channel names and output records from seq1.json:

channels: ["Device1/CH1", "Device1/CH5", "Device2/CH3", "Device2/CH5"]   # ids 1..4
outputs:
  { id:0, time_id:4, len:7,  val:8,  cond:0, chn:1 }    # → Device1/CH1, value node #8
  { id:1, time_id:5, len:10, val:11, cond:0, chn:3 }    # → Device2/CH3
  { id:5, time_id:6, len:13, val:15, cond:0, chn:2 }    # → Device1/CH5 (the ramp)
measures:
  { id:3, time_id:7, chn:4 }                            # → Device2/CH5
FieldMeaningExample (output 0)
chn1-based index into channels1 → Device1/CH1
valnode id of the value/ramp expression8
lennode id of the pulse length7
time_id1-based index into the times table4
condnode id of a gating condition; 0 = unconditional0

Every field lines up with the source: three add(...) calls became three output records on channels 1/3/2, and the addMeasure became the one measure on channel 4. The reader recovers the experiment's structure from raw bytes.

7. lib/seq_manager.py — the engine wrapper

A thin mirror of matlab_new/lib/SeqManager.m with two deliberate safety properties:

  • Lazy load. import seq_manager does not load libnacs. The DLL is imported only on the first get(), so the entire test suite can import freely without ever touching the engine.
  • Mutable buffer. create_sequence coerces to bytearray, because the ctypes binding uses ctypes.c_uint8.from_buffer, which requires a writable buffer — immutable bytes would raise.
def get():
    global _MANAGER
    if _MANAGER is None:
        from libnacs.expseq_manager import Manager   # imported lazily on purpose
        _MANAGER = Manager()
    return _MANAGER

def create_sequence(data):
    if not isinstance(data, bytearray):
        data = bytearray(data)        # ctypes.from_buffer needs writable memory
    return get().create_sequence(data)
Compile-only create_sequence only compiles the bytes; it never calls init_run/start, so it does not drive hardware. Even so, because it loads the engine on the shared PC, the test that uses it is opt-in (see next section).

8. MATLAB capture — engine-free by construction

tools/capture_matlab_reference.m generates new references without ever loading the engine or touching hardware. The trick is the codebase's own test hook:

SeqManager.override_tick_per_sec(1000);            # makes tick_per_sec() return WITHOUT loading libnacs
cleanup = onCleanup(@() SeqManager.override_tick_per_sec(0));
...
bytes = serialize(s);                              # serialize() ONLY — never generate()/run()
fwrite(fopen([name '.bin'],'w'), bytes, 'int8');

tools/reference_list.m supplies the sequences as {name, params, build} entries whose build handles mirror the proven TestExpSeq.m patterns (addStep(len).add('Device/CH', value), function-handle ramps, a global).

Run in a separate MATLAB session Because SeqManager is a singleton, run this in a different MATLAB than the one driving the experiment. With the override set, no engine loads and no device is touched — but never share the session that is mid-run.

10. Why this proves correctness (and what's next)

Round-tripping all six real serialize() outputs byte-for-byte proves two things at once: the byte-format spec in the appendix is correct, and the reader implements it faithfully — verified against genuine MATLAB output with no MATLAB run and no hardware.

That makes the reader the executable contract for Phase 1: when the Python SeqContext / serialize() starts producing bytes, we assert they decode() to the same structure (or match byte-for-byte) against these very references, and diff pinpoints any divergence down to the field.
