Phase 2 walkthrough — Sequence tree & timing

pyctrl Python front-end migration · the builder that turns an ExpSeq tree into the serialized byte blob

Goal of Phase 2: build sequences in Python — time steps, pulses, ramps, sub-sequences, branches, ordering constraints — whose serialize() output is a byte array identical to what MATLAB produces for the equivalent sequence. This is the engine that everything downstream feeds: Phase 3 gives it the real config so production YbSeqs can be built; Phase 4 will sweep it.

This doc walks the class tree, the two-level byte format, the timing model, and the three places where a byte hinges on something non-obvious.

TL;DR

  • THE ONE RULE: serialize() must equal MATLAB's byte-for-byte. Phase 2 is a straight 1:1 transliteration of matlab_new/lib/ — the brassboard re-architecture ideas (a new EventTime model, a SeqCompiler/BackendBase split, a "new" BasicSeq) were all declined after reading the actual source; see the design note.
  • TestExpSeq test1–test7 reproduce the committed matlab_new/lib/test/seq{1..6}.json byte-for-byte, plus ramp / timing / branch / TTL-manager cases and reader round-trips over the real YbSeqs corpus.
  • Build + serialize() benchmarked ~8–10× faster than MATLAB R2023a (~5 µs/output, linear; negligible next to a shot).
  • Runs entirely no_hardware — the engine is never loaded; the check is pure byte math.
  • Status: DONE & committed — 654d101. Phase 3 (config & globals) is built directly on this layer.

The byte format — two levels

The MATLAB↔engine boundary is a single blob. ExpSeq.serialize() (exp_seq.py:146) writes the top level; each basic sequence is written by RootSeq.serialize_bseq() (root_seq.py:115). All packing is little-endian (struct '<').

Top level

[version = 0 : 1B]
[nnodes : 4B][ node records … ]         # seq_ctx.node_serialized()
[nchns  : 4B][ chnname\0 … ]            # compacted cid count (disabled -> 0, skipped)
[ndefvals : 4B][ chnid:4B, type:1B, value … ]   # skip default==0 / disabled
[nslots : 4B][ … ]                      # globals (seq_ctx.global_serialized())
[nnoramp : 4B = 0]
[nbasicseqs : 4B][ BasicSeq … ]         # ExpSeq is #1, then self.basic_seqs
[ndatas : 4B][ … ]                      # interpolation tables (data_serialized())
[nbackenddatas : 4B][ ZYNQZYNQ blocks … ]

Ordering subtlety that costs bytes: the basic sequences are built first (exp_seq.py:191-194) because walking them is what interns the value nodes into the SeqContext. Only then is node_serialized() emitted — into its slot near the front of the blob (exp_seq.py:202). So the node table is positioned early but computed late; build order and emit order differ on purpose.

Per basic sequence (serialize_bseq)

[ntimes : 4B][ sign:1B, id:4B, delta_node:4B, prev_id:4B … ]   # the SeqTime chain
[nendtimes : 4B][ time_id:4B … ]
[ntimeorders : 4B][ sign:1B, id:4B, before_id:4B, after_id:4B … ]
[noutputs : 4B][ id:4B, time_id:4B, len:4B, val:4B, cond:4B, chn:4B … ]
[nmeasures : 4B][ id:4B, time_id:4B, chn:4B … ]    # count includes skipped (disabled) ones
[nassigns : 4B][ assign_id:4B, global_id:4B, val:4B … ]
[nbranches : 4B][ branch_id:4B, target_id:4B, cond:4B … ][ default_target : 4B ]

The timing model — a signed linked list, not an offset

SeqTime (seq_time.py:35) is a point in time represented as a chain of additive terms, each carrying its own id and sign:

class SeqTime:
    UNKNOWN = 0; NONNEG = 1; POS = 2          # serialized as a 1-byte int8 sign
    def __init__(self, seq, id, sign, parent, term):
        ...   # `term` is already in TICKS; build only via zero()/create()/combine()

Each term is serialized per-term as [sign:1B][id:4B][delta_node:4B][prev_id:4B]. That is why pyctrl declined brassboard-seq's EventTime{static|rt} "single offset" model: collapsing the chain into one offset slot would lose those per-term bytes. add_order(sign, before, after) records an ordering constraint; add_equal is simply two add_order(NONNEG, …) calls (root_seq.py:110-112) — there is no OrderEqual opcode.

Three places a byte hinges on something non-obvious

The exp_seq_base.py module docstring says it plainly: "Every byte hinges on (a) the global obj_counter interleaving …"

  1. Creation-order interleaving. Pulse / time / order / measure / assign / branch ids are all drawn from one counter — SeqContext.next_obj_id() (seq_context.py:130). So the Python build must call into the framework in MATLAB's exact creation order; a re-ordered .add() or add_order shifts every downstream id and the bytes diverge. (This is why the ported steps are faithful call-for-call, not "logically equivalent".)
  2. Rounding mode. MATLAB round is half-away-from-zero, applied at every round(t * time_scale) site. Python's built-in round is banker's rounding and would silently drift. pyctrl routes all time rounding through mat_utils.mat_round (mat_utils.py:22): 
    def mat_round(x):
    if isinstance(x, SeqVal): return x.__round__()   # -> OP_RINT node
    if x >= 0: return float(math.floor(x + 0.5))
    return float(math.ceil(x - 0.5))
  3. int vs float constants (status quo). A bare Python number used as a channel value serializes as float64, mirroring a MATLAB bare double — so real steps saturated with .add('TTL…', 1) / .add('Amp…', 0) match byte-for-byte with zero int/float work. "Option 2" (int-natural 3 = int32) stays deferred/unbuilt until a real sequence forces it. (Phase 3 later confirmed this empirically across the whole real corpus.)

Branches & multiple basic sequences

ExpSeq.new_basic_seq() creates a BasicSeq (a branch target); a parent routes into it with cond_branch(cond, target) (and an optional default branch). The target's bseq_id is what lands in the [nbranches] table (root_seq.py:163+); each basic sequence serializes the same per-bseq layout. This is a plain transliteration of the existing MATLAB BasicSeq.m/ConditionalWrapper.m — naming it did not let us delete a tree-walker (there isn't one: the C++ engine only ever consumes create_sequence(bytes)). The real RearrangeCommSeq/RearrangeCommSeq2 sequences (2- and 3-basic-seq) exercise this path in Phase 3.

Design note — what was declined from brassboard-seq

The original plan floated three brassboard-inspired re-architectures for Phase 2. An 8-way read of the actual lib/ classes settled them — transliterate, don't re-architect:

ProposedVerdictWhy
EventTime{static|rt} + TimeOrder enumdeclinedSeqTime is a signed linked list whose terms serialize individually; one offset slot loses bytes.
SeqCompiler + BackendBase splitdeclinedOnly one backend (libnacs) exists; the byte layout already lives in RootSeq.serialize_bseq. No seq_compiler.py / libnacs_backend.py built.
introduce a "new" BasicSeqsupersededBasicSeq/ConditionalWrapper are existing MATLAB classes — a 1:1 port, with no SeqManager tree-walker to delete.

The byte format is sourced only from MATLAB (via capture_matlab_reference.m); brassboard's 1 ps EventTime and rtval AST look wire-compatible but target a different engine (ARTIQ/RFSoC) — never copy its bytes.

Performance

Build + serialize() is ~8–10× faster than MATLAB R2023a (~5 µs per output, linear in sequence size — negligible next to an experimental shot). The one Python hotspot is inspect.signature in TimeStep's pulse-arity check (~25% of build for function-pulse-heavy sequences); caching _nargin is a deferred optimization, batched with the end-of-migration perf passes.
