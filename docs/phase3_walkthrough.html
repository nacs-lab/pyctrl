Phase 3 walkthrough — Config & globals

pyctrl Python front-end migration · work items W1–W5

Goal of Phase 3: give pyctrl the real experiment config (expConfig consts, channel aliases, default values), the MATLAB obj.field(default) access model (DynProps/SubProps/Consts), and a hand-ported step cone — enough that real YbSeqs sequences can be built in Python and serialize byte-identical to the MATLAB capture.

This doc walks the five work items (W1–W5), the locked design decisions, the non-obvious byte-equality fixes, and where the migration stands.

TL;DR

  • THE ONE RULE still governs: for every sequence, serialize() must produce a byte array identical to MATLAB's. Phase 3 is the first time real production sequences (not synthetic TestExpSeq) are reconstructed end-to-end in Python and byte-checked.
  • 15 of the 16 ok-corpus YbSeqs build byte-identical to tests/reference_ybseqs/ybseqs_reference.json — the full linear family plus both RearrangeComm branch sequences (2- and 3-basic-seq). (EOM616Ramp is deliberately excluded — see W5.)
  • Config is a committed JSON capture (tests/reference/config_reference.json), opt-in via SeqConfig.load_real(); the default stays an empty config so Phase-2 byte tests are untouched.
  • Full suite: 284 passed / 13 deselected, no_hardware (engine never loaded).
  • Status: W1–W5 done (W6 was folded into W2). Only W7 (globals round-trip, needs_engine) remains before Phase 4 (ScanGroup).

W1 — SeqConfig (config loader)

lib/seq_config.py is a process singleton that mirrors how MATLAB picks the config by which expConfig.m is on the path:

  • Empty by default (SeqConfig.get()): no aliases (identity channel translation), no consts, no defaults — so the Phase-2 seq*.json byte tests keep building unchanged.
  • Real config opt-in: SeqConfig.load_real() loads the committed capture. A real-seq build must call load_real() first and SeqConfig.reset() in teardown, or the real config leaks into other tests (it's a process singleton).

What the loader reproduces from SeqConfig.m:

  • translate_channel — recursive alias expansion (only the first /-component is aliased, then re-joined and re-resolved), trailing-/ trim done at capture, a name_map memo with a None loop-sentinel.
  • default_vals re-keyed by translated backend name (with MATLAB's conflict check).
  • check_channel_disabled → False (disabledChannels is empty in expConfig.m).

W2 — DynProps + SubProps (the obj.field(default) model)

A faithful 1:1 transliteration of DynProps.m (chosen over the brassboard ParamPack triple — the access tree is zero bytes, so the triple bought nothing for byte-equality while breaking the byte-mismatch→MATLAB-method debugging map).

  • DynProps owns the value store and the access tracker; SubProps is a thin path-carrying proxy that re-enters the root on every read/write (so a = c.X; a.f = 1 mutates c.X.f — handle-class aliasing, reproduced deliberately).
  • Semantics: scalar-leaf read → value and marked accessed; struct read → SubProps, not marked; parens c.X(default) → resolve + mark + persist the default (struct defaults merge, existing/leftmost wins, NaN-as-missing); brace c.X[default] → proxy, not marked, still persists; dp() collapses accessed to the True sentinel.
  • tests/test_dyn_props.py is full TestDynProps.m parity (9 tests) — this is also W6 (brace {}→__getitem__, test_disp rewritten).

W3 — Consts()

lib/consts.py is the whole 8-line port:

class Consts(DynProps):
    def __init__(self):
        super().__init__(SeqConfig.get().consts)   # DynProps deep-copies the store

Consts() is a DynProps over the active config's consts — it wraps the same tree ExpSeq copies into s.C, so a bare-build step reads identical values via both Consts().X and s.C.X. The W2 deepcopy makes every Consts() instance independent.

W4 — the step cone + seq wiring (the capstone)

The first time real production sequences are built in Python. Layout mirrors matlab_new/:

  • YbSteps/<StepName>.py — 16 steps the ok corpus actively calls: backbone InitStep/BlueMOTStep/SLMStep/GreenMOTStep/LACStep/Imag399Step + CoreShellMOTStep/AbsImagStep + BlueLACStep/Cool556Step/Cool556hXStep + SLMTrapModulationStep/ReleaseRecaptureStep + PushoutStep/Pushout399Step/PushouthXStep. Each is def <StepName>(s, g) — a faithful transliteration of the add/add_step/wait/ramp_to call sequence (byte output depends on call order — an internal counter interleaves).
  • YbSeqs/<SeqName>.py — 15 sequences wiring the steps. Deferred server/SLM/camera callbacks (regBeforeStart/regBeforeBSeq/regAfterEnd) → no-op stubs; serialize() never runs them.
  • Naming: YbSteps/ and YbSeqs/ keep the MATLAB names verbatim (InitStep.py / def InitStep), not snake_case — they mirror the .m filenames 1:1 and are referenced by those names across scan files. (The snake_case transliteration rule still governs lib/ only.) Both dirs are on the flat import path (pyproject.toml + tests/conftest.py).
  • tests/test_ybseqs_build.py — builds each ok seq (real config, tick 1e12) and asserts serialize() == the capture byte for byte, plus repeatability.

How it was built: a 3-step CoreShell cone was ported and byte-validated first to lock the conventions, the remaining 13 steps were ported in parallel and adversarially verified line-by-line against their MATLAB source, and the byte oracle (all 15 seq builds) was the final arbiter — green on the first validated pass.

W5 — finalize the corpus + int/float canary

  • EOM616Ramp dropped from the byte-build corpus (→ 15/16). It is the lone ok seq that reads MemoryMap at build time, so its bytes depend on lab state, not the sequence definition. pyctrl deliberately does not model MemoryMap — see the design note below. It uses no step-cone steps, so coverage is complete.
  • test_seq_has_no_int32_constants (capture-independent): decode each built seq and assert zero int32 constants. Empirically confirms the int→float64 mapping across the real corpus, so option-2 int32 coercion stays unbuilt.

Three non-obvious byte-equality fixes

Each silently breaks byte-equality if missed (all landed in W1/W2):

  1. Eager-resolve proxy default args. Consts().A.B.C as a default argument is a bare SubProps in Python (MATLAB resolves it to a number first). DynProps._make_default resolves any SubProps/DynProps arg to its value — else a proxy is written into the store and corrupts every later read.
  2. Deepcopy the nested store in DynProps.__init__. A shallow dict(store) aliases nested dicts shared with SeqConfig.consts; default-writes would mutate shared consts.
  3. Float-coerce every numeric consts/default leaf on JSON load. JSON 1→Python int would tag ARG_CONST_INT32 and diverge from MATLAB's double→ARG_CONST_FLOAT64.

The only SeqVal-operand case in the corpus is the Pushout-family EOM616 block: abs((Freq - global) * 20e-9 * 3.0) + 20e-3 — the 3 had to be written 3.0 (MATLAB double→FLOAT64; a bare int → INT32 and diverges).

One related transliteration trap: in seq bodies, a bare scalar-leaf read like Consts().Imag399.Amp must be forced with () before use in .add(...) — MATLAB auto-resolves a scalar leaf, but pyctrl's SubProps is lazy and would otherwise reach .add() unresolved. The steps avoid this because they always use the g.X(Consts()...) default-call form.

Design note — no MemoryMap in pyctrl

For byte-equality the storage mechanism is irrelevant — only the value baked into the bytes matters, so a MemoryMap abstraction in the front-end buys nothing and couples the build to lab state. The byte-reproducible pattern (already used by the Pushout seqs): declare a sequence global at build (s.new_global()) and let the deferred runtime callback inject the persisted value via set_global — the build stays pure, the global's value is set at runtime. EOM616Ramp is the legacy outlier that reads MemoryMap directly at build; revisit it only if pyctrl grows a real runtime (Phase 5), where cross-shot state belongs in an explicit store (JSON/SQLite), not a memory-mapped file.
